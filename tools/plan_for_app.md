# Plan: Streamlit Web App for TTW Register Tools

## Context

The Green Party Brent team has CLI tools (`clean_register.py`, `enrich_register.py`, `validate_enrichment.py`) for converting council electoral register data into TTW Digital upload format. These work well but require Python and command-line knowledge. The goal is to wrap them in a Streamlit web app so other non-tech-savvy Green Party local groups can use them via a browser.

**User requirements**: Green Party audience, Streamlit Cloud deployment, guided wizard pipeline, in-memory data only (no persistence), built-in help, friendly branding.

## File layout

```
tools/webapp/
    app.py                 # Streamlit entry point
    bridge.py              # tempfile bridge to CLI scripts
    test_bridge.py         # Unit tests for bridge + report parsing
    report_parser.py       # Parse QA reports into structured sections
    election_config.py     # Predefined elections + auto historic/future
    help_content.py        # Built-in guide text
    requirements.txt       # streamlit>=1.30.0,<2.0
    .streamlit/
        config.toml        # Green Party theming
```

Existing scripts (`clean_register.py`, `enrich_register.py`, `validate_enrichment.py`, `ttw_common.py`) remain **unchanged**.

## Strategy: tempfile + subprocess bridge

Rather than refactoring the scripts (which have `sys.exit()` in 16+ places and tightly integrated file I/O), use a thin bridge:

1. Write uploaded `BytesIO` to temp files
2. Call scripts via `subprocess.run(cmd, capture_output=True, text=True, cwd=TOOLS_DIR)` with `--quiet`
3. Read output CSV + report + any auto-generated files (unmatched CSV) back as in-memory bytes/text
4. Temp directory auto-cleaned by context manager
5. On non-zero exit code, capture `stderr` and display via `st.error()`

**Path resolution**: All subprocess calls use `cwd=TOOLS_DIR` (resolved as `Path(__file__).resolve().parent.parent`) so the scripts' `sys.path.insert(0, Path(__file__).parent)` imports of `ttw_common` work correctly.

This means any CLI updates automatically flow through to the app with zero maintenance.

## UI: 3-step wizard

Single-page app with session-state-gated steps:

### Step 1: Clean Register
- File upload for DS1 (council register CSV)
- Election picker (predefined multiselect dropdown, auto-assigns historic/future by date)
- Checkbox: "Input has enrichment columns (GE24, Party, 1-5)" → maps to `--enriched-columns`
- Mode auto-inferred: elections selected → `register+elections`, otherwise → `register`
- Optional: date format dropdown (DMY default), strip-extra, strip-empty checkboxes
- **Output**: Summary metrics, interactive QA report (expandable sections), download cleaned CSV

### Step 2: Enrich
Three optional file uploaders (at least one required):
- **Enriched register** (register-format CSV with historic election data) → `--enriched-register`
- **Canvassing export** (TTW canvassing export with profile_name/address) → `--canvassing-export`
- **Canvassing register** (register-format with future election data) → `--canvassing-register`

Election names auto-populated from Step 1 selections, split into:
- `--historic-elections` (elections where date < today)
- `--future-elections` (elections where date >= today)

Match threshold slider (default 0.8)

**Output**: Match statistics, interactive QA report, download enriched CSV + unmatched CSV (if generated)

**Bridge note**: After running enrich, check for auto-generated `{output_stem}.unmatched.csv` in temp dir and capture it.

### Step 3: Validate
- Auto-runs using Step 1 output (base) and Step 2 output (enriched)
- Also passes: `--report` (enrichment QA report path), `--canvassing-export` (DS3 if provided), `--unmatched` (if generated by Step 2)
- **Important**: validate_enrichment prints its report to stdout (does not write a file). Bridge captures stdout as the validation report text.
- **Output**: PASS/WARN/FAIL badges per check, download report text
- "Download All" zip button at bottom (all CSVs + all reports)

Steps 2 & 3 greyed out until prerequisite completes.

## Bridge specifications

### `run_clean(uploaded_bytes, elections, election_types, options)`
- Writes uploaded bytes to `tmpdir/input.csv`
- Builds cmd: `[python3, clean_register.py, input, output, --report, report_path, --quiet, ...]`
- If elections selected: adds `--mode register+elections --elections ... --election-types ...`
- If enriched columns checked: adds `--enriched-columns`
- Always passes explicit `--report` to known temp path
- Returns: `{success, output_csv_bytes, report_text, stderr}`

### `run_enrich(base_csv_bytes, sources, elections, threshold)`
- `sources` dict: `{enriched_register: bytes|None, canvassing_export: bytes|None, canvassing_register: bytes|None}`
- Writes each non-None source to temp files
- Builds cmd with appropriate `--enriched-register`, `--canvassing-export`, `--canvassing-register` flags
- Adds `--historic-elections` and `--future-elections` from `split_elections()`
- After run, checks for `{output_stem}.unmatched.csv` in temp dir
- Returns: `{success, output_csv_bytes, report_text, unmatched_csv_bytes|None, stderr}`

### `run_validate(output_csv_bytes, base_csv_bytes, enrich_report_text, canvassing_export_bytes, unmatched_csv_bytes, elections)`
- Writes all inputs to temp files
- Builds cmd: `[python3, validate_enrichment.py, output, --base, base, --report, enrich_report, ...]`
- If canvassing_export provided: adds `--canvassing-export`
- If unmatched provided: adds `--unmatched`
- Captures **stdout** as the validation report (script prints, doesn't write file)
- Returns: `{success, exit_code, report_text, stderr}`

All subprocess calls use `cwd=TOOLS_DIR` for correct import resolution.

## QA report rendering

Parse the machine-readable footer (`FIX|Row=...|Field=...` lines) into DataFrames for interactive display:

| Section | Rendering |
|---------|-----------|
| Summary | `st.metric()` widgets (input rows, output rows, fixes, warnings) |
| Fixes | Sortable DataFrame [Row, Field, Old, New, Reason] |
| Warnings | Table split into "NEEDS MANUAL FIX" (red) and others |
| Deletions | Table with elector details |
| Match results | Metrics + tables for confident/possible/ambiguous/unmatched |
| Validation | Color-coded PASS/WARN/FAIL badges |

Plus `st.download_button()` for the raw text report.

## Branding

- **App name**: "TTW Data Prep" (friendly, descriptive)

## Streamlit Cloud constraints

- **Upload limit**: 200MB default (sufficient for electoral registers)
- **Memory**: ~1GB free tier — fuzzy matching on large files (100k+ rows) may be tight
- **Timeout**: Long-running enrich steps may need a progress spinner/warning for large files
- **Python version**: Specify in `runtime.txt` if needed (3.9+)

## Implementation sequence

1. `bridge.py` — `run_clean()`, `run_enrich()`, `run_validate()` with full arg construction
2. `test_bridge.py` — unit tests using existing `test_data/` fixtures
3. `report_parser.py` — parse human + machine-readable report sections
4. `election_config.py` — predefined list + `split_elections()` function
5. `app.py` Step 1 only — clean pipeline end-to-end locally
6. `app.py` Steps 2 + 3 — enrich (3-source UI) + validate (stdout capture)
7. `help_content.py` — formatting rules, QA guide, TTW prerequisites
8. Polish — error handling, branding, edge cases
9. Deploy to Streamlit Cloud

- `run_enrich()` with all 3 source types → verify correct arg construction
